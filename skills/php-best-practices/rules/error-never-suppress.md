# Never Suppress Errors with `@`

## Why it matters
The `@` operator instructs PHP to discard any error generated by the expression, including warnings that indicate genuine failures. Code that would have triggered a useful error message instead returns `false` or `null` with no explanation. Debugging why a value is unexpectedly `false` in production, with no log entry and no stack trace, is expensive.

## Rule
Never use the `@` error suppression operator. Every failure must be handled explicitly, transformed into an exception, logged and rethrown, or allowed to propagate. Silent suppression destroys visibility.

## Bad
```php
<?php

declare(strict_types=1);

// Errors hidden — false/null returned with no diagnostic
$data    = @file_get_contents('/path/to/file');
$decoded = @json_decode($json);
$result  = @unserialize($payload);
$conn    = @mysqli_connect('localhost', 'user', 'pass');

// Empty catch is the same mistake with more syntax
try {
    $service->process($data);
} catch (\Exception $e) {
    // swallowed
}
```

## Better
```php
<?php

declare(strict_types=1);

// Return-value checks replace @ but still swallow context
$data = file_get_contents('/path/to/file');
if ($data === false) {
    return null; // caller gets null with no reason
}

$decoded = json_decode($json, true);
if (json_last_error() !== JSON_ERROR_NONE) {
    return null;
}
```

## Best
```php
<?php

declare(strict_types=1);

// File read: check readability, then throw on failure
function readFile(string $path): string
{
    if (!is_readable($path)) {
        throw new \RuntimeException("File not readable: {$path}");
    }

    $data = file_get_contents($path);

    if ($data === false) {
        throw new \RuntimeException("Failed to read file: {$path}");
    }

    return $data;
}

// JSON: validate then decode (PHP 8.3+)
function parseJson(string $json): mixed
{
    if (!json_validate($json)) {
        throw new \InvalidArgumentException('Invalid JSON: ' . json_last_error_msg());
    }

    return json_decode($json, true);
}

// PDO: exceptions on by default, so no suppression needed
$pdo = new \PDO($dsn, $user, $pass, [
    \PDO::ATTR_ERRMODE    => \PDO::ERRMODE_EXCEPTION,
    \PDO::ATTR_EMULATE_PREPARES => false,
]);

// Intentional skip: comment + log explain the decision
try {
    $mailer->send($notification);
} catch (MailerException $e) {
    // Notification email is non-critical; log and continue
    $logger->warning('Notification email failed', ['error' => $e->getMessage()]);
}
```

## Exceptions / trade-offs
There is no legitimate use of `@` in new code. Legacy codebases sometimes use it around third-party functions that emit notices — migrate those call sites to proper error handling rather than leaving the suppression in place.

## Static-analysis notes
PHPStan reports `@`-suppressed expressions as a code smell at level 5+. Psalm has `suppressAll` detection. Both can be configured to treat suppression as an error.

## Version notes
`json_validate()` — PHP 8.3+
